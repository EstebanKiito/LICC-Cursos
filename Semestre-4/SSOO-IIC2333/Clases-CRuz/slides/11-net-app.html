Arquitectura client-servidor/
Aplicaciones se construyen a través de procesos en distintos nodos que se comunican
  • Clientes se comunican con servidor
    • Clientes no se comunican entre ellos
  • Servidor posee una dirección fija y conocida
  • Servidor puede recibir múltiples solicitudes
    • Requerimientos intensivos en infraestructura
    • Capacidad de atención, ancho de banda, …
    • Suelen ubicarse en data centers
Web, FTP, Telnet, SSH, email, …
Arquitectura peer-to-peer (P2P)/
Aplicaciones se construyen en base a comunicación directa entre miembros (pares)
  • Requerimientos mínimos en existencia de servidores
  • Facilita auto-escalabilidad
    • Miembros añaden capacidades a la red
BitTorrent, download accelerators, telefonía por internet (Skype), IPTV
Desafíos
  • ISP Friendly
    • Muchas conexiones son asimétricas
  • Seguridad
    • Múltiples puntos de vulnerabilidad
  • Incentivos
    • ¿Por qué podría querer aportar recursos a la red?

Sockets

  • Programador puede controlar parámetros del socket
    • Tipo de servicio de transporte (TCP / UDP)
    • Direccionamiento del host remoto (IP + puerto)

Sockets

Una vez que ambos extremos, cliente y servidor, poseen un socket, es posible mandar mensajes.


Socket requiere definición de:
  • Tipo de servicio de transporte
    • UDP: comunicación rápida, no confiable (unreliable)
    • TCP: comunicación confiable
  • Dirección del extremo opuesto
    • Dirección IP remota
    • Puerto remoto. Cada socket está asociado a un puerto

Conexión UDP

Creación de socket con AF_INET, SOCK_DGRAM
int socket(int domain, int type, int protocol);

  • domain. Familia de protocolos. Ej: AF_LOCAL, AF_INET, AF_PACKET, …
  • type. Tipo de socket. Ej: SOCK_DGRAM, SOCK_STREAM
  • protocol. Protocolo. Default: 0. Se suele determinar según domain y type

Conexión UDP

Envío de mensaje UDP
ssize_t sendto(int sockfd, const void *buf, size_t len, int flags, const struct sockaddr *dest_addr, socklen_t addrlen);

  • sockfd. Socket obtenido de socket
  • buf. Buffer con datos a enviar
  • len. Longitud (byte) de los datos a enviar
  • flags. Flags.
  • dest_addr. struct con datos de destino.
  • addrlen. Tamaño de dest_addr
Retorno: cantidad de bytes enviados

Conexión UDP

Recepción de mensaje UDP
ssize_t recvfrom(int sockfd, void *buf, size_t len, int flags, struct sockaddr *src_addr, socklen_t *addrlen);

  • sockfd. Socket obtenido de socket
  • buf. Buffer con datos a enviar
  • len. Longitud (byte) de los datos a enviar
  • flags. Flags.
    • i src_addr. struct con datos de destino. Si es NULL, se ignora.
  • addrlen. Tamaño de src_addr
Retorno: cantidad de bytes recibidos

Conexión TCP

Establecimiento de conexión: TCP Handshake
  • Requiere establecer conexión antes de enviar
  • Cliente inicia solicitud de conexión en un puerto conocido del servidor
  • Servidor debe tener un socket en ese puerto
  • Servidor debe aceptar la conexión
  • Al aceptar la conexión, servidor asigna nuevo socket para la conexión

HTTP

HyperText Transfer Protocol

Un servicio de archivos remoto. RFC1945, RFC2616.
  • Modelo cliente-servidor
  • Cliente HTTP: solicita y recibe archivos
    • Opcionalmente podría parsearlos e interpretarlos localmente
  • Servidor HTTP: recibe solicitudes y envía archivos


HTTP define el formato de los mensajes
Acceso a URL. Conexión no-persistente.
http://iic2333.ing.puc.cl/slides/5-disk.html

  1. Cliente HTTP inicia conexión TCP a iic2333.ing.puc.cl en puerto 80. Socket en cliente, y socket en server.
  2. Cliente HTTP envía Request HTTP incluyendo ruta y archivo: slides/5-disk.html
  3. Servidor HTTP recibe Request HTTP, busca el archivo slides/5-disk.html, lo encapsula en Response HTTP, y envía al cliente a través de su socket
  4. Servidor HTTP indica a socket que cierre la conexión TCP.
  5. Cliente HTTP recibe Response HTTP y extrae el archivo. Examina el archivo HTML y encuentra referencias a otros archivos (por ejemplo, imágenes JPG, PNG)
  6. Cliente puede re-ejecutar los pasos 1 al 4, para cada archivo que necesite.


Servidores HTTP actuales usan conexiones persistentes y pipelining.

DNS

Domain Name System

Antes de iniciar la conexión TCP para el protocolo HTTP a URL:
http://iic2333.ing.puc.cl/slides/5-disk.html

  1. Cliente ejecuta proceso cliente DNS
  2. Browser extrae iic2333.ing.puc.cl y lo pasa al cliente DNS
  3. Cliente DNS envía query por iic2333.ing.puc.cl a servidor DNS
  4. Cliente DNS recibe respuesta de servidor DNS, que incluye IP de iic2333.ing.puc.cl
  5. Cliente DNS pasa respuesta a browser
  6. Cliente HTTP inicia conexión TCP a la IP de iic2333.ing.puc.cl


DNS utiliza una estructura jerárquica de dominos y servidores

Domain Name System

Base de datos distribuida y jerárquica
Servidores almacenan mappings
  • Si no conoce un mapping, lo consulta a un servidor superior en la jerarquía
  • Eventualmente puede llegar a servidor raíz
  • Servidores raíz conocen ubicación de servidores TLD
    • gTLD: General Top-Level Domain. .com, .org, .edu
    • ccTLD: Country-Code Top-Level Domain. .cl, .nl, .es, .au
  • 13 servidores raíz en el mundo: A,B,C,D,E,F,G,H,I,J,K,L,M
    • Con múltiples instancias de cada uno (cuatro de ellos replicados en Chile)
    • http://www.root-servers.org
  • Servidores que administran el mapping son servidores autoritativos
    • Servidores autoritativos administran el mapping (al nivel de institución)
    • Servidores no-autoritativos mantienen temporalmente el mapping
Consultas iterativas y recursivas
Consultas iterativas (1,2) envían respuesta (completa o parcial) al origen


Consultas recursivas (4,6) reenvían la consulta

Al resolver un mapping, éste es almacenado en los servidores
  • Caching de respuestas permite reducir el tráfico
  • Respuestas cached posee un timeout
  • Servidores TLD suelen estar en cache de servidores locales
  • Servidores raíz no son tan visitados
